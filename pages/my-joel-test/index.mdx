## Near instantaneous setup of a local development environment

This should resemble step 2 in Joel's test: "Can you make a build in one step" 

Joel's step is all about reducing the time and trouble to **ship your work**. I think it's even more important to reduce the time it takes to **start work**. Look around, and you'll see most open source projects make sure the "Getting started" section of every Readme is short. Oftentimes it's just two lines:

```shell script
yarn
yarn start
```


## Can you work on the frontend and backend for a feature simultaneously? 

We are all now familiar with the bifurcation of backend and frontend. And, according to the 2020 Stackoverflow developer survey, [55% of develpers self identify as full-stack](https://insights.stackoverflow.com/survey/2020#developer-profile-developer-type). And consequently, job listings specify "full-stack" positions - especially startups, which need to move fast and need small teams or even individual developers to build "full features".
 
So, much like the first rule above, this one is to make sure that working on backend and frontend functionality for a feature isn't a serial process. It shouldn't be "first the api, then later the ui".

Besides making it possible for one person to work on the full feature, it allows for more rapid iteration of UI and API. Face it, the API contracts are never perfect the first time, and UI enhancements are almost always requested in the early, dog-fooding phase of internal usability testing (or even [hallway usability testing](https://www.joelonsoftware.com/2000/08/09/the-joel-test-12-steps-to-better-code/)).


## Do you check PRs for lint errors?

This one may sound obvious - but you cannot take it for granted. More than once, I have been horrified to discover the number of lint errors in a codebase after accepting the job.

The number of lint errors is directly correlated to "how much love" was given to the codebase. You want to work on a codebase that the other developers have cared for enough to keep clean and consistent (enough).
